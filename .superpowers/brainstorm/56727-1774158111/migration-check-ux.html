Migration Check: Early vs In-Flow

Both catch the match before wasted effort — the question is where it feels most natural.

A: Check on /join page (before account)
🎓 I'm a UNAM graduate
Already have an account?
✅ Profile found!
Juan García — 7a Generación

Pros

  • Catches match earliest
  • No wasted signup steps
  • Can redirect to login

Cons

  • Exposes profile existence to anyone
  • User not authenticated yet — can't write merge request
  • Adds friction to /join for everyone
C: Check at Step 3 inline (during UNAM form)
Step 1: ✅ Account created
Step 2: ✅ Selected "Member"
Step 3: UNAM Verification
← check triggers on blur/submit
🔗 Existing profile detected!
We found a profile matching this número de cuenta.
You can claim it after registration — an admin will review.

Pros

  • User is authenticated — can write merge request
  • Natural flow — no extra page
  • Only members see it (no info leak)
  • Works with existing merge system

Cons

  • User already created account before seeing match
  • Slightly later in funnel

💡 Recommendation: Option C

Option A has a security problem: anyone can probe whether a numeroCuenta has a profile without being authenticated. Option C keeps the check behind authentication, integrates cleanly with the existing merge system (potentialMergeMatch is already built), and the "wasted" Steps 1-2 take about 30 seconds — minimal cost. The user gets a clear inline message right when they enter their numeroCuenta.
